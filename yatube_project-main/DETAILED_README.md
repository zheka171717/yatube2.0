# Создание проекта – социальная сеть Yatube

## Описание проекта (документ, который прислал заказчик)

**Yatube** - социальная сеть для публикации личных дневников.

Это сайт, на котором можно создать свою страницу. Если на неё зайти, то можно посмотреть все записи автора.

Пользователи могут заходить на чужие страницы, подписываться на авторов и комментировать их записи.

Автор может выбрать имя и уникальный адрес для своей страницы.

Для администратора есть возможность модерировать записи и блокировать пользователей, если начнут присылать спам.

Свои записи можно отправить в сообщество и посмотреть там другие записи от разных авторов.

Обычно веб-проекты работают на специальных серверах, но в ходе разработки проекта Yatube мы тестировали его, запуская прямо на рабочем компьютере. Для такого запуска в Django встроен упрощённый сервер, его называют «сервер разработчика», или «dev-сервер».

## 2. Создание приложений

### 2.1. Создание приложения posts

Можно создать приложение при помощи `manage.py`. Достаточно использовать в директории с `manage.py` команду:

```bash
python manage.py startapp posts
```

- `startapp` — команда создания шаблона приложения
- `posts` — название приложения

Существуют и другие команды, с помощью которых можно создать приложение:
- `django-admin startapp posts`
- `python -m django startapp posts`

### 2.2. Структура приложения

При создании приложения в директории создаётся базовая структура файлов, которая, по мнению разработчиков Django, пригодится для любого приложения:

- **`__init__.py`** — указывает интерпретатору Python, что текущий каталог — это пакет (его можно импортировать; если нет `__init__.py` — импорт может сработать некорректно).
- **`admin.py`** — здесь можно настроить отображение админ-зоны приложения.
- **`apps.py`** — настройки конфигурации приложения.
- **`models.py`** — здесь разработчик описывает устройство базы данных приложения.
- **`migrations/`** — тут хранится история изменений в базе данных.
- **`tests.py`** — файл для тестов приложения.
- **`views.py`** — тут хранятся обработчики запросов (функции или классы, получающие запрос и генерирующие ответ).

Эту структуру можно дополнять и улучшать.

### 2.3. Регистрация приложения

В каждом приложении автоматически создаётся файл `apps.py`, в нём описываются настройки приложения. В этом файле автоматически создаётся класс, конфигурирующий приложение; по умолчанию в нём создаётся лишь одно поле — `name`.

Чтобы Django-проект узнал о том, что в нём есть приложение, нужно это приложение зарегистрировать: добавить его конфиг в список установленных приложений `INSTALLED_APPS` в файле `settings.py`:

```python
INSTALLED_APPS = [
    'posts.apps.PostsConfig',  # Добавленная запись
    'ice_cream.apps.IceCreamConfig',  # Учебное приложение
    'django.contrib.admin',
    'django.contrib.auth',
    'django.contrib.contenttypes',
    'django.contrib.sessions',
    'django.contrib.messages',
    'django.contrib.staticfiles',    
]
```

Зарегистрировать приложение можно и иным способом: можно добавить в `INSTALLED_APPS` не конфиг, а имя приложения — `posts`. Но практичнее регистрировать через конфиг: ведь тогда в `apps.py` можно будет в любой момент добавить дополнительные настройки приложения. При регистрации через имя такой возможности не будет.

## 3. Работа с URLs и Views

### 3.1. Принцип работы

После того как Django найдёт совпадение запрошенного URL с шаблоном адреса из списка, должен быть вызван обработчик — функция или класс, в которых после обработки запроса будет подготовлен ответ.

Начиная с версии Django 2 для связи URL и обработчика применяется функция `path()`. Она принимает обязательные параметры `path('route', view)`:

- **`route`** — шаблон обрабатываемого адреса, образец, с которым сравнивается полученный запрос;
- **`view`** — функция-обработчик: если запрошенный URL совпадает с `route`, вызов будет перенаправлен в указанную view-функцию (view-функции в Django хранят в файле `views.py`).

Все `path()` хранят в файле `urls.py` в виде списка (обычно его называют `urlpatterns`).

### 3.2. Настройка urls.py для проекта

В проекте Yatube такие адреса:
- Главная страница
- Страницы сообществ (groups)

Мы настроили `urls.py` так, чтобы главная страница обрабатывалась вью-функцией `index()` из приложения `posts`. Для этого в головном файле `yatube/urls.py` добавили путь для обработки главной страницы:

```python
from django.contrib import admin
from django.urls import include, path

urlpatterns = [
    # импорт правил из приложения posts
    path('', include('posts.urls')),
    path('admin/', admin.site.urls),
]
```

**Что здесь происходит:**
- `path('', include('posts.urls'))` — говорит Django: если запрос пришёл на пустой путь (главная страница), передай обработку всех URL, начинающихся с этого пути, в файл `urls.py` приложения `posts`. Это называется "включение" (include) URL-конфигурации другого приложения.

### 3.3. Создание urls.py в приложении posts

В приложении `posts` создали файл `urls.py` и добавили в него код:

```python
from django.urls import path
from . import views

urlpatterns = [
    path('', views.index),
]
```

**Что здесь происходит:**
- `path('', views.index)` — если запрос пришёл на пустой путь (после того, как мы уже отсекли часть URL в главном `urls.py`), вызываем функцию `index` из файла `views.py`.

### 3.4. Настройка URL для групп

Аналогичным образом настроили обработку запросов к адресам вида `group/<slug:slug>/` (конвертер `slug` ожидает строку из букв и цифр, входящих в стандарт ASCII, а также символов `-` и `_`):

```python
from django.urls import path
from . import views

urlpatterns = [
    path('', views.index, name='index'),
    path('group/<slug:slug>/', views.group_posts, name='group_list'),
]
```

**Что здесь происходит:**
- `<slug:slug>` — это конвертер пути. Он "захватывает" часть URL и передаёт её в view-функцию как аргумент с именем `slug`. Например, при запросе `/group/cats/` в функцию `group_posts` будет передан аргумент `slug='cats'`.
- `name='group_list'` — даёт этому маршруту имя, чтобы мы могли ссылаться на него в шаблонах через `{% url 'posts:group_list' slug='cats' %}`.

### 3.5. Создание view-функций

Первоначально view-функции для страниц проекта Yatube (`yatube/posts/views.py`) выглядели так:

```python
from django.shortcuts import render
from django.http import HttpResponse

def index(request):
    """Главная страница Yatube"""
    return HttpResponse(
        '<h1>Главная страница Yatube</h1>'
        '<p>Добро пожаловать в нашу уютную социальную сеть!</p>'
        '<p>Здесь вы найдете:</p>'
        '<ul>'
        '<li>Интересные посты от пользователей</li>'
        '<li>Сообщества по интересам</li>'
        '<li>Много общения и полезной информации</li>'
        '</ul>'
        '<p><a href="/group/funny-cats/">Посмотреть посты про смешных котов</a></p>'
        '<p><a href="/group/programming/">Зайти в сообщество программистов</a></p>'
    )

def group_posts(request, slug):
    """Страница с постами, отфильтрованными по группам"""
    return HttpResponse(
        f'<h1>Сообщество: {slug}</h1>'
        f'<p>Вы находитесь в группе <strong>"{slug}"</strong></p>'
        f'<p>Здесь будут отображаться посты, относящиеся к этому сообществу.</p>'
        f'<h2>Последние посты в группе:</h2>'
        f'<ul>'
        f'<li>Пост №1: "Как я вступил в группу {slug}"</li>'
        f'<li>Пост №2: "10 причин любить группу {slug}"</li>'
        f'<li>Пост №3: "Мой первый пост в {slug}"</li>'
        f'</ul>'
        f'<p><a href="/">Вернуться на главную</a></p>'
    )
```

**Что здесь происходит:**
- Каждая view-функция принимает объект `request` (содержит информацию о запросе) и возвращает объект `HttpResponse` (содержит HTML-код, который будет отправлен пользователю).
- В функции `group_posts` мы принимаем дополнительный параметр `slug` — именно его Django "захватил" из URL.
- Пока что мы просто возвращаем "захардкоженный" HTML, но позже заменим это на настоящие шаблоны.

## 4. Работа с шаблонами (templates)

### 4.1. Настройка хранения шаблонов на уровне проекта

Далее мы настраивали проект Yatube так, чтобы хранить шаблоны (HTML-код выносят в отдельные файлы, их и называют HTML-шаблоны) на уровне проекта.

В корне проекта создали пустую папку `templates`. Внутри папки создали директорию `posts` и разместили в ней файл `index.html`:

```html
<!-- templates/posts/index.html -->
<!DOCTYPE html>
<html lang="ru">
  <head>
  </head>
  <body>
    <h1>Главная страница</h1>
  </body>
</html>
```

В файле `yatube/settings.py` добавили настройки для шаблонов:

```python
TEMPLATES_DIR = os.path.join(BASE_DIR, 'templates')

TEMPLATES = [
    {
        'BACKEND': 'django.template.backends.django.DjangoTemplates',
        'DIRS': [TEMPLATES_DIR],  # Говорим Django искать шаблоны в этой папке
        'APP_DIRS': True,  # Также искать в папках templates внутри приложений
        'OPTIONS': {
            'context_processors': [
                'django.template.context_processors.debug',
                'django.template.context_processors.request',
                'django.contrib.auth.context_processors.auth',
                'django.contrib.messages.context_processors.messages',
            ],
        },
    },
]
```

**Что здесь происходит:**
- `'DIRS': [TEMPLATES_DIR]` — добавляем путь к нашей папке `templates` в корне проекта. Django будет искать шаблоны в этом каталоге.
- `'APP_DIRS': True` — также искать шаблоны в папках `templates` внутри каждого приложения.

С помощью функции `render()` подключили шаблон к соответствующей view-функции. Файл `yatube/posts/views.py` стал выглядеть так:

```python
from django.shortcuts import render

def index(request):
    """Главная страница Yatube"""
    template = 'posts/index.html'  # Путь к шаблону
    return render(request, template)  # Рендерим шаблон и возвращаем результат

def group_posts(request, slug):
    """Страница с постами, отфильтрованными по группам"""
    template = 'posts/group_list.html'
    context = {
        'slug': slug,  # Передаём slug в шаблон
    }
    return render(request, template, context)
```

**Что здесь происходит:**
- `render(request, template, context)` — это сокращённая форма. Вместо того чтобы загружать шаблон через `loader.get_template()`, а потом вызывать `template.render()`, мы делаем всё одной функцией.
- `context` — словарь с данными, которые мы хотим передать в шаблон. В шаблоне мы сможем обращаться к ним по ключам.

Потом мы запускали проект в dev-режиме и проверяли, что шаблон доступен по адресу `http://127.0.0.1:8000/`.

### 4.2. Разбиение шаблонов на логические блоки

После мы разбивали шаблоны на логические блоки с помощью тегов `{% extends %}`, `{% block %}` и `{% include %}`.

Обычно шаблоны не пишут целиком в одном файле: их разделяют на части и сохраняют в отдельных файлах.

При подготовке ответа возвращаемая HTML-страница монтируется из этих фрагментов, как из конструктора Lego, и в собранном виде отправляется клиенту.

Для такой практики есть несколько веских причин:

- Как правило, страницы проекта содержат повторяющиеся фрагменты: шапка сайта, подвал, блок навигации. Карточки товаров в интернет-магазине или анонсы статей на информационном ресурсе тоже свёрстаны одинаково. Эти элементы в одинаковом виде присутствуют на многих страницах. Код каждого такого элемента сохраняют в отдельный файл, а при рендеринге встраивают их в общий шаблон. В этом тоже реализация принципа DRY: если, например, потребуется изменить шапку сайта, изменения вносятся в один файл. В ином случае пришлось бы редактировать десятки шаблонов, исправляя шапку в каждом из них.
- Шаблон для нового типа страницы можно подготовить за считанные минуты, собрав его из готовых фрагментов, как из конструктора.

#### Тег `{% include %}`

Повторяющиеся части шаблонов можно вынести в отдельные файлы и по необходимости подключать к любой странице. Для подключения применяют тег `{% include 'адрес_подключаемого шаблона' %}`. Подключаемые шаблоны обычно хранят в специальной папке `includes/`.

#### Тег `{% block %}`

Дочерние шаблоны могут менять информацию в блоках, описанных в родительском шаблоне, например:

```html
{% block content %}
    Контент не подвезли :(
{% endblock %}
```

**Что здесь происходит:**
- `{% block content %}` — объявляет блок с именем "content". Дочерние шаблоны могут переопределить содержимое этого блока.
- `Контент не подвезли :(` — это содержимое по умолчанию, которое будет использовано, если дочерний шаблон не переопределит блок.

#### Тег `{% extends %}`

В остальных шаблонах указывается, что они являются дочерними для базового шаблона (или «расширяют» его: англ. extend — «расширять»):

```html
{% extends 'base.html' %}
```

**Что здесь происходит:**

### 4.5. Namespace и name для URL

Чтобы ссылки в шаблонах не были жёстко заданы и автоматически подстраивались под изменения в `urls.py`, мы использовали `namespace` и `name`:

## 5. Работа с базами данных и моделями

### 5.1. Добавление модели в проект

Для хранения постов пользователей нам потребовалась модель (с помощью моделей разработчик определяет структуру таблиц, их поля и взаимосвязи между ними). В Django код моделей хранят в файлах `models.py` в каждом приложении.

В файл `posts/models.py` мы добавили код модели `Post`:

```python
from django.db import models
from django.contrib.auth import get_user_model

User = get_user_model()  # Получаем модель пользователя (встроенную в Django)

class Post(models.Model):
    text = models.TextField()  # Текстовое поле для поста
    pub_date = models.DateTimeField(auto_now_add=True)  # Дата публикации (устанавливается автоматически)
    author = models.ForeignKey(  # Связь с моделью User
        User,
        on_delete=models.CASCADE,  # Если пользователь удалён — удаляем и его посты
        related_name='posts'  # Позволяет обращаться из User ко всем его постам: user.posts.all()
    )
```

**Что здесь происходит:**
- `models.TextField()` — поле для хранения большого текста.
- `models.DateTimeField(auto_now_add=True)` — поле даты и времени, которое автоматически заполняется текущим временем при создании записи.
- `models.ForeignKey` — создаёт связь "многие к одному". Один автор может иметь много постов.
- `on_delete=models.CASCADE` — если автор будет удалён, все его посты тоже удалятся (чтобы не было "сиротских" записей).
- `related_name='posts'` — создаёт обратную связь. Теперь у объекта `user` появится атрибут `user.posts`, через который можно получить все посты этого пользователя.

### 5.2. Миграция данных

После создания модели нужно было создать таблицы в базе данных. Для этого мы использовали механизм миграций:

**Что здесь происходит:**
- `makemigrations` — анализирует изменения в моделях и создаёт файлы миграций в папке `migrations/`. Эти файлы содержат инструкции по изменению структуры БД.
- `migrate` — выполняет все неприменённые миграции, фактически изменяя структуру базы данных.

### 5.6. Вывод информации из базы данных на страницы

Мы изменили view-функции, чтобы они получали данные из БД:

**`posts/views.py`:**
```python
from django.shortcuts import render
from .models import Post  # Импортируем модель

def index(request):
    # Получаем 10 последних постов (сортировка по убыванию даты)
    posts = Post.objects.order_by('-pub_date')[:10]
    context = {
        'posts': posts,  # Передаём посты в шаблон
    }
    return render(request, 'posts/index.html', context)
```

**Что здесь происходит:**
- `Post.objects` — это менеджер запросов к базе данных для модели `Post`.
- `.order_by('-pub_date')` — сортирует по полю `pub_date` в обратном порядке (минус означает убывание).
- `[:10]` — ограничивает результат 10 записями (аналог `LIMIT 10` в SQL).
- Всё вместе — один запрос к базе данных, но написанный на Python, а не на SQL.

**Шаблон `index.html` был обновлён для вывода постов:**
```html
{% for post in posts %}  <!-- Перебираем все посты -->
  <article>
    <ul>
      <li>Автор: {{ post.author.get_full_name }}</li>  <!-- Получаем полное имя автора -->
      <li>Дата публикации: {{ post.pub_date|date:"d E Y" }}</li>  <!-- Форматируем дату -->
    </ul>
    <p>{{ post.text }}</p>  <!-- Выводим текст поста -->
    {% if post.group %}  <!-- Если у поста есть группа -->
      <a href="{% url 'posts:group_list' post.group.slug %}">все записи группы</a>
    {% endif %}
  </article>
  {% if not forloop.last %}<hr>{% endif %}  <!-- Разделитель между постами -->
{% endfor %}
```

**Что здесь происходит:**
- `{{ post.author.get_full_name }}` — обращаемся к связанному объекту автора и вызываем его метод `get_full_name()`.
- `{{ post.pub_date|date:"d E Y" }}` — применяем фильтр `date` к дате для форматирования. `d E Y` означает: день, полное название месяца, год (например, "30 июля 1854").
- `{% if post.group %}` — проверяем, есть ли у поста группа (некоторые посты могут быть без группы).
- `{% if not forloop.last %}` — `forloop.last` — это специальная переменная цикла, которая равна `True` на последней итерации. Ставим разделитель только между постами, но не после последнего.

```python
class Group(models.Model):
```

### 6.6. Обновление view-функции group_posts

Изменили функцию для отображения постов конкретной группы:

```python
from django.shortcuts import render, get_object_or_404  # Новый импорт
from .models import Post, Group

def group_posts(request, slug):
    # Пытаемся найти группу с таким slug, если нет — возвращаем 404
    group = get_object_or_404(Group, slug=slug)
    
    # Получаем посты только этой группы
    posts = Post.objects.filter(group=group).order_by('-pub_date')[:10]
    
    context = {
        'group': group,  # Передаём объект группы
        'posts': posts,
    }
    return render(request, 'posts/group_list.html', context)
```

**Что здесь происходит:**
- `get_object_or_404(Group, slug=slug)` — это удобная функция. Она пытается получить объект из базы данных по указанным критериям. Если объект не найден, автоматически возвращает страницу с ошибкой 404 (Not Found).

### 7.2. Добавление пагинации (постраничной навигации)

Для удобства просмотра большого количества постов была добавлена пагинация:

**Что такое пагинация?**  
Пагинация (pagination) — это механизм разбиения длинного списка записей на отдельные страницы. Вместо того чтобы загружать сразу все посты (что может быть медленно и неудобно), мы показываем только 10 постов на одной странице.

#### Реализация в views.py:

```python
from django.core.paginator import Paginator  # Импортируем встроенный класс пагинации

def index(request):
    """Главная страница Yatube с пагинацией"""
    # Получаем все посты, отсортированные по дате
    post_list = Post.objects.order_by('-pub_date')
    
    # Создаем пагинатор: 10 постов на страницу
    paginator = Paginator(post_list, 10)
    
    # Получаем номер страницы из GET-параметра (например, ?page=2)
    page_number = request.GET.get('page')
    
    # Получаем объекты нужной страницы
    page_obj = paginator.get_page(page_number)
    
    context = {
        'page_obj': page_obj,  # Передаем в шаблон объект страницы
    }
    return render(request, 'posts/index.html', context)
```

**Как это работает:**
1. `Paginator(post_list, 10)` — создаёт объект пагинатора, который разбивает все посты на страницы по 10 штук
2. `request.GET.get('page')` — извлекает из URL параметр `page` (например, `/posts/?page=2`)
3. `paginator.get_page(page_number)` — возвращает объект `Page` с постами для запрошенной страницы
4. Если номер страницы не указан, автоматически показывается первая страница
5. Если указан несуществующий номер, показывается последняя страница

#### Шаблон пагинатора (includes/paginator.html):

Мы создали отдельный файл для пагинации, который можно подключать на любую страницу:

```html

    {% if page_obj.has_previous %}

      </li>
```

## 9. Реализация поиска по тексту постов

### 9.1. Что сделано
- Добавлен поиск по тексту постов на главной странице
- Реализована подсветка найденных слов жёлтым фоном

### 9.2. Код и объяснения

**views.py:**
```python
def index(request):
    # Получаем все посты, сортируем по дате (новые сверху)
    post_list = Post.objects.all().order_by('-pub_date')
    
    # Получаем поисковый запрос из адресной строки (например, ?q=утро)
    query = request.GET.get('q', '')
    
    # Если пользователь что-то ищет - фильтруем посты
    if query:
        # icontains - поиск по части слова без учёта регистра
        post_list = post_list.filter(text__icontains=query)
    
    # Важно! select_related подгружает связанные данные (автора и группу)
    # сразу одним запросом, чтобы потом не делать отдельный запрос для каждого поста
    # Это решает проблему N+1 запросов
    post_list = post_list.select_related('author', 'group')
    # ... пагинация и контекст
```

**Для чего это нужно:**  
Без `select_related` при обращении к `post.author` в шаблоне Django делал бы отдельный запрос к базе для каждого поста. Если на странице 10 постов - это 10 лишних запросов. `select_related` объединяет всё в один запрос через JOIN, что значительно ускоряет загрузку страницы.

**Пользовательский фильтр (templatetags/post_extras.py):**
```python
@register.filter
def highlight(text, query):
    """Подсвечивает найденные слова в тексте"""
    # re.escape - экранирует спецсимволы, чтобы можно было искать например "$"
    escaped_query = re.escape(query)
    # IGNORECASE - игнорирует регистр (утро = Утро = УТРО)
    pattern = re.compile(f'({escaped_query})', re.IGNORECASE)
    # Оборачиваем найденные слова в span с классом highlight
    highlighted = pattern.sub(r'<span class="highlight">\1</span>', str(text))
    # mark_safe - разрешает выводить HTML (без него теги бы экранировались)
    return mark_safe(highlighted)
```

**CSS для подсветки:**
```css

    background-color: #ffeb3b;  /* Ярко-жёлтый фон */
    color: #000;                 /* Чёрный текст для контраста */
    font-weight: bold;           /* Жирный текст */
    padding: 0 2px;              /* Немного воздуха вокруг слова */
    border-radius: 3px;          /* Скруглённые уголки */
}
```

**Шаблон (index.html):**
```html

<!-- Выводим текст поста с применением фильтра подсветки -->

```

**Для чего это нужно:**  
Фильтр применяется прямо в шаблоне к тексту поста, автоматически подсвечивая все вхождения поискового запроса. Если запроса нет - фильтр ничего не меняет.

```python
    query = request.GET.get('q', '')
    groups = Group.objects.all()
    
    if query:
        groups = groups.filter(
            Q(title__icontains=query) | Q(description__icontains=query)
```

### 12.3. Декораторы

**Что такое декоратор:**  
Декоратор - это функция, которая принимает другую функцию и возвращает её изменённую версию. Позволяет добавлять функциональность без изменения исходного кода.

**Встроенный декоратор @login_required:**
```python
from django.contrib.auth.decorators import login_required

@login_required
def post_create(request):
    """Страница создания нового поста (только для авторизованных)"""
    # Только авторизованные пользователи могут создавать посты
    form = PostForm(request.POST or None)
    if request.method == 'POST' and form.is_valid():
        post = form.save(commit=False)
        post.author = request.user  # Автор - текущий пользователь
        post.save()
        return redirect('posts:index')
    return render(request, 'posts/post_form.html', {'form': form})
```

**Для чего это нужно:**  
- Декоратор `@login_required` проверяет, авторизован ли пользователь
- Если пользователь не авторизован - перенаправляет на страницу входа
- Позволяет не писать проверку `request.user.is_authenticated` в каждой view-функции

### 12.5. Страница создания и редактирования постов

```python
def profile_search(request):
    """Поиск пользователей по имени или username"""
    query = request.GET.get('q', '')
    users = User.objects.all()
    
    if query:
        users = users.filter(
            Q(username__icontains=query) | 
            Q(first_name__icontains=query) | 
            Q(last_name__icontains=query)
        ).order_by('username')
    
    paginator = Paginator(users, 12)
    page_number = request.GET.get('page')
    page_obj = paginator.get_page(page_number)
    
    context = {
        'page_obj': page_obj,
        'query': query,
        'total_users': users.count(),
    }
    return render(request, 'posts/profile_search.html', context)
```
